# Workflow (SDD — Spec-Driven Development)

Один цикл = одна фича. Запрещено переходить к следующему этапу без явного подтверждения готовности текущего.

---

## Общие правила

- Запрещено переходить к следующему этапу, пока есть неразрешённые вопросы или неоднозначности; при их обнаружении — сначала задать уточняющие вопросы
- Допущения разрешены только явно и отдельным списком
- Запрещены «молчаливые» допущения
- При обнаружении пробела на любом этапе — возврат на тот этап, где пробел должен был быть закрыт, с повторным ревью затронутых артефактов

### Уровни важности замечаний

Используются на этапах ревью (2, 4, 6):

- **критический** — блокирует переход к следующему этапу; без исправления цикл не продолжается
- **высокий** — блокирует переход; исправляется до завершения этапа
- **средний** — желательно исправить, но не блокирует; фиксируется и может быть отложено
- **низкий** — косметика, на усмотрение

---

## 1. Бриф

Источник: GitHub issue.

Содержание:

- описание проблемы (что нужно / что сломано)
- зачем это нужно (ценность для пользователя/бизнеса)
- **Границы** — что входит в фичу
- **Вне границ** — что сознательно не делаем в рамках этой фичи
- **Критерии приёмки** — проверяемые утверждения вида «при X система делает Y»
- **Допущения** — если есть
- **Открытые вопросы** — если есть

Критерий завершения:

- каждый критерий приёмки приведён к форме «при X система делает Y» или эквивалентной, допускающей однозначную проверку
- все критические неоднозначности либо устранены, либо переведены в открытые вопросы

---

## 2. Ревью брифа

- проверка на ясность
- поиск недостающей информации
- поиск противоречий
- группировка вопросов: продуктовые / технические / UX-UI

Критерий завершения:

- отсутствуют критические и высокие замечания
- критерии приёмки утверждены

---

## 3. Спецификация

Полная техническая спецификация на основе утверждённого брифа.

Должна включать:

- архитектурные решения
- модели данных (если применимо), включая стратегию для существующих данных
- поведение UI на уровне логики (если применимо)
- обработку edge cases

Критерий завершения:

- все вопросы, вскрывшиеся при написании спецификации, разрешены или возвращены на этап 1 через обновление брифа
- все критерии приёмки адресованы

---

## 4. Ревью спецификации

Запуск: `/spec-reviewer:spec-review`

Критерий завершения:

- отсутствуют критические и высокие замечания

---

## 5. План реализации

Детальный пошаговый план.

Должен включать:

- изменения backend
- изменения frontend (Phlex + Tailwind)
- миграции (если нужны)
- зависимости / гемы (если нужны)
- **тестовая стратегия** — что тестируется на уровне unit / integration / system и что сознательно не покрыто тестами (с обоснованием)

Требования к каждому шагу:

- явный вход и выход
- побочные эффекты (если есть)
- затрагиваемые файлы
- способ проверки шага

Критерий завершения:

- план можно выполнить без дополнительной интерпретации
- тестовая стратегия присутствует

---

## 6. Ревью плана

Проверка:

- полнота
- edge cases
- пропущенные шаги
- потенциальные риски
- корректность тестовой стратегии

Критерий завершения:

- отсутствуют критические и высокие замечания
- план детерминирован и безопасен для выполнения

---

## 7. Реализация

Выполнение плана пошагово.

Правила:

- отклонения, меняющие контракт шага (вход, выход, побочные эффекты, затрагиваемые файлы, способ проверки), требуют явного согласования
- мелкие решения внутри шага (имена переменных, внутренний порядок методов, форматирование) согласования не требуют
- при появлении неизвестности действует общее правило возврата на соответствующий этап
- изменения небольшие и проверяемые; после каждого шага — проверка, описанная в плане

---

## 8. Приёмка

Короткий финальный чек:

- все критерии приёмки из брифа выполнены
- тесты из тестовой стратегии проходят
- полный тестовый набор проекта проходит (отсутствие регрессий)

Если чек пройден — формируется PR.

Описание PR должно включать:

- ссылку на исходный issue
- краткое саммари изменений (что сделано и зачем)
- чек-лист критериев приёмки с отметками о выполнении
- инструкции для ручной проверки (если применимо)

Ревью и мерж PR — ответственность автора цикла (не агента).

Критерий завершения цикла:

- чек пройден
- PR сформирован
- PR смержен
- цикл закрыт явно
