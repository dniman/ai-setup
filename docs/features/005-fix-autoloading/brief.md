# Бриф: Исправление автозагрузки Components::Base и Views::Base

**Issue:** #5

## Что делаем

Исправляем конфликт автозагрузки Zeitwerk, из-за которого `Views::Base` и компоненты в `app/views/components/` не загружаются.

Сейчас `app/views/` зарегистрирована как autoload path с namespace `Views` (в `config/initializers/phlex.rb`). Zeitwerk ожидает, что `app/views/components/base.rb` определит `Views::Components::Base`, но файл определяет `Components::Base`. В результате:

- `Views::Components::Base` — ошибка: "expected file to define constant, but didn't".
- `Components::Base` — не загружается, т.к. нет autoload path для этого namespace.
- `Views::Base` наследует от `Components::Base` и падает с `NameError`.

Нужно привести namespace классов в соответствие с ожиданиями Zeitwerk, сохранив две отдельные иерархии наследования:

- **Компоненты:** базовый класс компонентов -> `Phlex::HTML`
- **Вью:** `Views::Base` -> базовый класс компонентов -> `Phlex::HTML`

Компоненты наследуются от своего базового класса, вью — от `Views::Base`. Это разные ветки, смешивать их не нужно.

Итоговый namespace базового класса компонентов (`Components::Base` vs `Views::Components::Base`) — см. открытые вопросы.

## Зачем

Сейчас все Phlex-вью вынуждены наследоваться напрямую от `Phlex::HTML` вместо `Views::Base`, дублируя include'ы (`Phlex::Rails::Helpers::Routes`, development-комментарии). `Views::Base` и `Components::Base` — ключевые базовые классы, без которых невозможно единообразное поведение вью и компонентов (общие хелперы, кеширование, отладочные комментарии).

## Scope

**Входит:**
- Исправление конфигурации автозагрузки в `config/initializers/phlex.rb` и/или namespace классов так, чтобы `Views::Base` и базовый компонент загружались без ошибок
- Сохранение работоспособности `Phlex::Kit` для модуля `Components`
- Обновление существующих вью, которые обходят баг (наследуются от `Phlex::HTML` вместо `Views::Base`), на наследование от `Views::Base`
- Удаление дублирующихся include'ов из вью после перевода на `Views::Base`

**Не входит:**
- Создание новых компонентов
- Изменение логики существующих вью и компонентов (только смена базового класса и удаление дублей)
- Рефакторинг лейаутов (они наследуются от `Phlex::HTML` по своей архитектуре)
- Перевод компонентов на наследование от `Views::Base` — компоненты наследуются от своего базового класса (`Components::Base`), не от `Views::Base`

## Критерии приёмки

- При загрузке `Views::Base` в Rails-приложении — класс загружается без ошибок
- При загрузке базового класса компонентов из `app/views/components/` — класс загружается без ошибок
- При наследовании вью от `Views::Base` — вью получает хелперы и development-комментарии из базового компонента
- При наследовании вью из authentication (#3) от `Views::Base` вместо `Phlex::HTML` — вью работает корректно
- При рендеринге существующих лейаутов — лейауты загружаются и рендерятся без ошибок (без изменений в их коде)
- При запуске `bundle exec rspec` — все существующие тесты проходят

## Открытые вопросы

- Какой итоговый namespace у базового класса компонентов: `Components::Base` или `Views::Components::Base`? Решение определяется на этапе спецификации при выборе варианта исправления автозагрузки.
